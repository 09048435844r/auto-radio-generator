# リサーチデータ生成パイプラインの現状分析

**作成日**: 2026-03-22  
**目的**: 「冗長ではなく、情報密度が極めて高い長尺のラジオ台本」を生成するための大元となる「リサーチデータ」の品質向上を目指し、現在のリサーチパイプラインの全貌を分析・ドキュメント化する。

---

## 1. 処理フローの全体像

### 1.1 エントリーポイント（UI → 関数呼び出し）

ユーザーがリサーチを実行する経路は2つ存在します：

#### **経路A: リサーチのみ実行**
```
UI: 「🔍 リサーチのみ実行」ボタン
  ↓
app.py: research_only_btn.click()
  ↓
app.py: research_only(theme, research_mode)
```

#### **経路B: 全自動実行（動画生成）**

> **Note (2026-04-23 更新)**: v3.7 で台本生成が SSOT 化され、Gradio 自動モードも
> 内部で `services.pipeline.execute_scripting_phase` に委譲される。リサーチ部分の
> みが `workflow._execute_gradio_scripting_phase` 内に残存。

```
UI: 「🎬 動画を生成」ボタン
  ↓
app.py: generate_btn.click()
  ↓
app.py: generate_video(...)
  ↓
workflow.py: execute_workflow(...) / run_workflow_sync(...)
  ↓
workflow.py: execute_planning_phase(...)
  ↓
workflow.py: _execute_gradio_scripting_phase(...)  ← リサーチ専用ラッパー
  ├─ PerplexityResearcher.research_multi(...)
  └─ services.pipeline.execute_scripting_phase(...)  ← SSOT な台本生成
```

### 1.2 リサーチ実行の詳細フロー

**経路A（リサーチのみ）の場合:**

```
1. app.py: research_only()
   ├─ Step 0: AIプロデューサーが検索計画を作成
   │   └─ GeminiClient.create_research_plan(theme, mode)
   │       ├─ システムプロンプト: _build_research_plan_prompt(mode)
   │       ├─ ユーザープロンプト: "テーマ: {theme}"
   │       └─ 出力: ResearchPlan { queries: [...], angle: "..." }
   │
   ├─ Step 1: 並列リサーチ実行
   │   └─ PerplexityResearcher.research_multi(queries, mode)
   │       ├─ 各クエリに対して並列API呼び出し（asyncio.gather）
   │       ├─ システムプロンプト: config.yaml の modes[mode].system_prompt
   │       ├─ ユーザープロンプト: "{query}\n\n上記について、背景・数字・事例・影響を含めて可能な限り詳細に、長文で解説してください。最低でも500文字以上の詳しい説明をお願いします。"
   │       └─ 出力: ResearchResult { content: "...", sources: [...] }
   │
   └─ Step 2: JSONL形式で保存
       └─ data/research/research_{timestamp}_{theme}.jsonl
```

**経路B（全自動）の場合:**

```
workflow.py: execute_planning_phase()
  └─ GeminiClient.create_research_plan() （同上）

workflow.py: _execute_gradio_scripting_phase()   ← v3.7 より薄い委譲ラッパー
  ├─ PerplexityResearcher.research_multi() （経路Aと同じ）
  ├─ _save_research_results() → research_brief.json を session 直下に保存
  └─ services.pipeline.execute_scripting_phase(research_brief, session_manager, config, ...)
      ├─ ScriptOrchestrator（TopicCurator → ShowRunner → SegmentGenerator）
      │   もしくは fallback の IScriptGenerator.generate()
      ├─ VisualPaletteGenerator.generate_palette() → VisualIdentity（dynamic BG 時）
      └─ MetadataGenerator.generate() → title / description / hashtags / thumbnail_title
```

## 2. 現在のプロンプトの内容（最重要）

### 2.1 検索計画作成プロンプト（Gemini API）

**関数**: `GeminiClient._build_research_plan_prompt(mode)`  
**ファイル**: `services/script_generation/gemini_client.py:514-601`

#### **ベースプロンプト**
```
あなたはラジオ番組の構成作家です。
テーマを面白く深掘りするための「検索クエリ」を考案してください。

## タスク
テーマに対して、以下の3つの観点から検索クエリを作成してください：
```

#### **モード別の指示（5種類）**

##### 1. **trivia（トリビア）**
```
1. **意外な事実**: あまり知られていない数字や歴史的背景を探るクエリ
   - 例: "○○の意外な歴史", "○○にまつわる驚きの数字"

2. **雑学・トリビア**: 面白いエピソードや豆知識を探るクエリ
   - 例: "○○の面白いエピソード", "○○の豆知識"

3. **深掘り**: 一般には語られない裏話や詳細を探るクエリ
   - 例: "○○の裏側", "○○の詳しい仕組み"
```

##### 2. **debate（ディベート）**
```
1. **賛成意見**: メリットや推進派の主張を探るクエリ
   - 例: "○○のメリット", "○○を支持する理由"

2. **反対意見**: デメリットや批判的な意見を探るクエリ
   - 例: "○○のデメリット", "○○に対する批判"

3. **対立軸**: 賛否両論や議論のポイントを探るクエリ
   - 例: "○○をめぐる議論", "○○の賛否両論"
```

##### 3. **weekly_digest（今週のまとめ）**
```
1. **最近の出来事**: 直近1ヶ月以内のニュースや動向を探るクエリ（「最近」「最新」など広義の言葉を使用）
   - 例: "○○ ニュース 最近", "○○ latest news 1 month", "○○ 最新動向 2024"
   - 注意: 「今週」「this week」などの厳密な期間指定は避ける（ニュースが見つからない可能性があるため）

2. **世間の反応**: SNSや世論の最近の反応を探るクエリ
   - 例: "○○ 最近の反応", "○○ 最新の評判", "○○ SNS 話題"

3. **今後の展望**: 最近発表された予測や今後の動きを探るクエリ
   - 例: "○○ 今後の予測", "○○ 最新の見通し", "○○ 将来性"
```

##### 4. **voices（世間の声）**
```
1. **口コミ・評判**: 実際の利用者の声や評価を探るクエリ
   - 例: "○○ 口コミ", "○○ 評判"

2. **体験談**: 実際の体験や事例を探るクエリ
   - 例: "○○ 体験談", "○○ 使ってみた"

3. **街の声**: SNSやフォーラムでの反応を探るクエリ
   - 例: "○○ SNS反応", "○○ みんなの意見"
```

##### 5. **lecture（解説・講座）**
```
1. **基本定義**: 初心者向けの分かりやすい説明を探るクエリ（専門用語を避ける）
   - 例: "○○ とは 初心者向け わかりやすく", "○○ 簡単に説明"

2. **仕組み・構造**: 図解や比喩を使った説明を探るクエリ
   - 例: "○○ 仕組み 図解 例え話", "○○ どういう仕組み 分かりやすく"

3. **メリット・活用例**: 何ができるか、どう役立つかを探るクエリ
   - 例: "○○ メリット 何ができる", "○○ 5歳児でもわかる", "○○ 具体例"
```

#### **出力形式**
```json
{
  "queries": [
    "検索クエリ1",
    "検索クエリ2",
    "検索クエリ3"
  ],
  "angle": "今回の台本の切り口・コンセプト（1文で）"
}
```

## 4. 現状の課題分析（考察）

### 4.1 情報量不足の根本原因

#### **問題1: クエリの抽象度が高すぎる**

**現状のクエリ例**:
- "○○の意外な歴史"
- "○○の面白いエピソード"
- "○○の裏側"

**問題点**:
- 抽象的すぎて、Perplexityが「どの程度の深さ」を求めているか判断できない
- 「意外な」「面白い」などの主観的な表現では、AIは表層的な情報を返しがち
- 具体的な数字・事例・専門家の見解などを明示的に要求していない

#### **問題2: システムプロンプトの指示が弱い**

**現状のシステムプロンプト（trivia）**:
```
あなたは雑学の専門家です。与えられたテーマについて以下を調査してください：
- 意外と知られていない事実（5つ以上）
- 歴史的な背景や由来
- 関連する面白いエピソード
- 専門家でも意見が分かれるポイント
結果は日本語で、「へぇ〜」と言いたくなるような形式で出力してください。
```

**問題点**:
- 「5つ以上」という数の指定はあるが、**各事実の深さ・詳細度**の要求がない
- 「へぇ〜と言いたくなる」という主観的な指示では、AIは短文の羅列で満足してしまう
- **具体的な情報密度の要求がない**（例: 各事実に対して「背景・数字・事例・影響」を含めるなど）

#### **問題3: max_tokensの制限が中途半端**

**現状**: `max_tokens: 8192`

**問題点**:
- 8192トークンは約6000-8000文字程度（日本語）
- 3つのクエリを並列実行すると、1クエリあたり約2000-2500文字
- これは「圧倒的な情報量」には不足している
- Perplexityは`max_tokens`に達すると途中で切れるため、**意図的に短くまとめてしまう可能性**がある

#### **問題4: ユーザープロンプトの詳細要求が不十分**

**現状**:
```
上記について、背景・数字・事例・影響を含めて可能な限り詳細に、長文で解説してください。最低でも500文字以上の詳しい説明をお願いします。
```

**問題点**:
- 「500文字以上」は最低ラインとしては低すぎる（1クエリあたり2000-3000文字を目指すべき）
- 「可能な限り詳細に」という曖昧な指示では、AIは適当なところで切り上げる
- **具体的な情報の種類**を列挙していない（例: 統計データ、専門家の引用、歴史的経緯、技術的詳細、実例など）

#### **問題5: 「避けてほしい話題」がリサーチに反映されない**

**現状**: `avoid_topics_input` はUIに存在するが、リサーチフェーズでは使用されていない。

**問題点**:
- ユーザーが「前回話した内容」を除外したくても、リサーチ段階で同じ情報が収集されてしまう
- 台本生成時に除外しても、リサーチデータ自体が重複しているため、台本の多様性が損なわれる

### 4.2 情報密度を高めるための改善アプローチ

#### **改善案1: クエリの具体化・構造化**

**現状のクエリ**:
```
"○○の意外な歴史"
```

**改善後のクエリ例**:
```
"○○の歴史的発展: 起源から現在までの主要なマイルストーン、技術的ブレークスルー、社会的影響、統計データ、専門家の評価を含む詳細な年表"
```

**効果**:
- 具体的な情報の種類を明示することで、AIが何を返すべきか明確になる
- 「詳細な年表」などの構造化要求により、情報の網羅性が向上する

#### **改善案2: システムプロンプトの強化**

**改善後のシステムプロンプト例（trivia）**:
```
あなたは雑学の専門家です。与えられたテーマについて、以下の要件を満たす詳細なレポートを作成してください：

## 必須要件
- **情報密度**: 各事実について、最低300文字以上の詳細な説明を含めること
- **具体性**: 抽象的な表現を避け、具体的な数字・日付・人名・組織名を含めること
- **多角性**: 技術的側面・歴史的側面・社会的側面・経済的側面など、複数の観点から分析すること
- **引用**: 可能な限り、専門家の見解・研究結果・統計データを引用すること

## 調査項目
1. **意外と知られていない事実（5つ以上）**
   - 各事実について: 背景、発見の経緯、具体的な数字、専門家の評価、社会的影響を含める
   
2. **歴史的な背景や由来**
   - 時系列で整理し、主要なマイルストーンを具体的な年月日とともに記述
   - 各時期の技術的・社会的背景を詳述
   
3. **関連する面白いエピソード**
   - 単なる逸話ではなく、そのエピソードが示す本質的な意味や影響を分析
   
4. **専門家でも意見が分かれるポイント**
   - 各立場の主張と根拠を詳細に説明
   - 最新の研究動向や議論の焦点を含める

## 出力形式
- 各セクションは見出しで明確に区切る
- 箇条書きと段落を適切に組み合わせる
- 専門用語には簡潔な説明を付ける
- 合計文字数: 最低3000文字以上を目標とする
```

**効果**:
- 情報密度の具体的な基準（300文字/事実、合計3000文字以上）を設定
- 「具体性」「多角性」「引用」などの質的要件を明示
- 出力形式を構造化することで、情報の整理が容易になる

#### **改善案3: max_tokensの拡大**

**現状**: `max_tokens: 8192`

**改善案**: `max_tokens: 16384` または `32768`

**効果**:
- 1クエリあたり4000-6000文字の詳細なリサーチが可能になる
- Perplexityが途中で切り上げる必要がなくなり、より深い情報を提供できる

**注意点**:
- コスト増加の可能性（Perplexityの料金体系を確認する必要あり）
- レスポンス時間の増加（ユーザー体験への影響を考慮）

#### **改善案4: ユーザープロンプトの詳細化**

**現状**:
```
上記について、背景・数字・事例・影響を含めて可能な限り詳細に、長文で解説してください。最低でも500文字以上の詳しい説明をお願いします。
```

**改善後**:
```
上記について、以下の要素を**すべて含めた**詳細なレポートを作成してください：

1. **背景・文脈**: 歴史的経緯、技術的前提、社会的背景
2. **具体的な数字・統計**: 市場規模、成長率、ユーザー数、技術指標など
3. **実例・事例**: 具体的な企業名、製品名、プロジェクト名、成功/失敗事例
4. **影響・インパクト**: 業界への影響、社会的影響、経済的影響、技術的影響
5. **専門家の見解**: 研究者、業界関係者、アナリストの具体的な発言や評価
6. **最新動向**: 直近6ヶ月以内のニュース、発表、トレンド
7. **議論のポイント**: 賛否両論、未解決の課題、今後の展望

**最低文字数**: 2000文字以上
**推奨文字数**: 3000-4000文字

可能な限り詳細に、マニアックに、深く掘り下げてください。表層的な情報ではなく、専門家レベルの深い洞察を求めています。
```

**効果**:
- 7つの具体的な要素を列挙することで、AIが何を含めるべきか明確になる
- 最低文字数を2000文字に引き上げることで、情報量が大幅に増加
- 「専門家レベルの深い洞察」という明示的な要求により、質が向上

#### **改善案5: 「避けてほしい話題」のリサーチへの統合**

**実装方針**:

1. `research_only()` 関数に `avoid_topics` パラメータを追加
2. `PerplexityResearcher.research_multi()` に `avoid_topics` を渡す
3. システムプロンプトまたはユーザープロンプトに除外指示を追加

**改善後のユーザープロンプト例**:
```python
if avoid_topics and avoid_topics.strip():
    detailed_query = f"""{query}

上記について、以下の要素をすべて含めた詳細なレポートを作成してください：
（省略）

## 除外事項
以下の話題は既に取り上げたため、**絶対に含めないでください**：
{avoid_topics}

上記以外の新しい情報・視点を提供してください。
"""
```

**効果**:
- リサーチ段階で重複を排除することで、台本の多様性が向上
- ユーザーの意図（前回と違う内容を作りたい）が正確に反映される

#### **改善案6: 多段階リサーチの導入**

**現状**: 1回のAPI呼び出しで完結

**改善案**: 2段階リサーチ
1. **第1段階**: 広範囲の情報収集（現状と同じ）
2. **第2段階**: 第1段階の結果から「深掘りすべきポイント」をAIが選定し、追加リサーチ

**実装イメージ**:
```python
# 第1段階: 広範囲リサーチ
initial_result = await researcher.research_multi(queries, mode)

# 第2段階: AIが深掘りポイントを選定
deep_dive_queries = await script_generator.suggest_deep_dive_queries(
    initial_result.content, 
    theme
)

# 第3段階: 深掘りリサーチ
deep_dive_result = await researcher.research_multi(deep_dive_queries, mode)

# 統合
final_result = merge_research_results(initial_result, deep_dive_result)
```

**効果**:
- 初回リサーチで見つかった「面白そうなポイント」をさらに深掘りできる
- 情報の深さと広さの両立が可能になる

**注意点**:
- API呼び出し回数が増加（コスト・時間の増加）
- `max_requests_per_workflow` の上限を考慮する必要あり

## 6. まとめ

3. **長期的な改善**: 多段階リサーチの導入
